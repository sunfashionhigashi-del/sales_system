# Git / GitHub 運用ルール（販売管理ソフト開発用）

本プロジェクトでは、「変更履歴の細かい管理」と「安全な機能保護」を最優先とし、コードの消失・意図しない変更・巻き戻し不能な状態を防ぎます。

## ■ 1. ブランチ運用ルール
* `main`ブランチでは直接作業を行わないこと。
* すべての作業は以下の形式でブランチを作成して行うこと。
  * `feature/機能名`
  * `fix/修正内容`
* 新しい作業を開始する際は、必ず新しいブランチを作成すること。

## ■ 2. コミットルール（最重要）
* 「作業の区切り」ではなく「意味のある変更単位」でコミットすること。
* 1つの機能追加・1つの修正ごとにコミットを作成すること。
* コミットメッセージは必ず変更内容が分かるように記述すること。
  * 良い例：「受注一覧表示を追加」「売上登録のバリデーション修正」「CSV出力の列順を修正」
  * 禁止：「修正」「更新」「調整」など曖昧な表現

## ■ 3. プッシュルール
* 以下のタイミングで必ずプッシュを行うこと：
  1. 機能単位の作業が完了したとき
  2. 大きな変更を行った直後
  3. セッション終了時
* プッシュ前には必ずコミットが適切に分割されているか確認すること。

## ■ 4. 機能保護ルール
* 完成した機能は必ず以下のいずれかで保護すること：
  1. `main`ブランチにマージする
  2. 安定版としてタグを付与する（例：`v0.1_受注登録完成`, `v0.2_売上機能安定版`）
* 既存機能を変更する場合は、必ず新しいブランチで作業すること。

## ■ 5. 変更影響の制御（重要）
* 変更対象の機能以外のコードには絶対に影響を与えないこと。
* 既存機能を削除・改変する場合は、明示的な指示がある場合のみ実施すること。
* 影響範囲が広がる可能性がある場合は、必ず事前に確認を行うこと。

## ■ 6. セッションログの扱い
* セッションログ（`session_log.md`など）は「補助情報」として扱う。
* 実際の復元はGitのコミット履歴を前提とすること。
* セッションログ更新時にも、対応するコード変更は必ずコミットすること。

## ■ 7. 禁止事項
* `main`ブランチへの直接コミット
* コミットせずに大規模変更を行うこと
* 既存機能の無断削除・無断改変
* 意味のないコミットの連続

## ■ 8. 作業前チェック（毎回実施）
* 現在のブランチ名
* 作業対象の機能範囲
* 影響を与えてはいけない既存機能

## ■ 9. 作業完了時チェック（毎回実施）
* 変更内容が機能単位でコミットされているか確認
* コミットメッセージが適切か確認
* 必要に応じてプッシュを実施

## ■ 最終原則
「いつでも任意の時点に戻れること」
「既存機能が意図せず壊れないこと」
を最優先とし、細かく明確なコミットを最重要ルールとする。
